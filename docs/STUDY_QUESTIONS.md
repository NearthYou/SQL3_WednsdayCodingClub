# 프로젝트 학습 질문 모음

## 1) 프로젝트를 보고 나올 수 있는 질문 10개

1. 왜 `team_1_wei`를 베이스로 하지 않고 `codex/api-mvcc`를 베이스로 잡았나요?
2. `/api/v1/sql`에만 캐시를 적용하고 `/batch`, `/page`에는 적용하지 않은 이유는 무엇인가요?
3. 캐시 무효화를 `TTL`만 쓰지 않고 `table version + TTL` 이중 조건으로 둔 이유는 무엇인가요?
4. `keep-alive` 도입 후 기존 단발성 요청 처리 대비 가장 크게 달라진 코드 경로는 어디인가요?
5. `/api/v1/metrics`에서 기존 필드는 유지하고 새 섹션만 추가한 이유는 무엇인가요?
6. `timing_ns(parse/lock_wait/engine/json/send)`를 실제 성능 개선에 어떻게 활용할 계획인가요?
7. `API_THR`, `DB_THR` 미지정 시 `cpu * 2`를 기본값으로 선택한 근거는 무엇인가요?
8. 현재 구조에서 가장 병목이 될 가능성이 높은 구간은 어디라고 보나요?
9. `test_api.sh`가 환경 제약(소켓 bind) 때문에 실패할 수 있는 점을 어떻게 CI에서 다룰 건가요?
10. 다음 단계에서 구조적으로 확장한다면 캐시 범위 확장과 MVCC 고도화 중 무엇을 먼저 할 건가요?

## 2) 개념 관련 질문 10개

1. MVCC는 무엇이고, lock 기반 동시성 제어와 어떤 차이가 있나요?
2. snapshot isolation은 어떤 일관성을 보장하고 어떤 문제(예: write skew)는 남길 수 있나요?
3. copy-on-write 방식의 장점과, write-heavy 환경에서의 단점은 무엇인가요?
4. keep-alive가 HTTP 성능에 주는 이점은 무엇이며, 어떤 리스크를 함께 가져오나요?
5. 캐시에서 TTL 기반 무효화와 버전 기반 무효화의 차이는 무엇인가요?
6. API 계층에서 관측성(observability) 지표를 설계할 때 핵심 원칙은 무엇인가요?
7. thread pool에서 queue 크기(`qmax`)와 worker 수를 결정할 때 어떤 트레이드오프가 있나요?
8. 읽기 중심 워크로드와 쓰기 중심 워크로드에서 최적 아키텍처가 어떻게 달라지나요?
9. "호환성을 유지한 API 확장"을 설계할 때 깨지기 쉬운 포인트는 무엇인가요?
10. 성능 최적화에서 "측정 가능한 지표"가 없는 상태에서의 개선 시도가 왜 위험한가요?
